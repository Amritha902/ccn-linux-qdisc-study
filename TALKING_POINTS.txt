SLIDE 1 — TITLE
"Good morning. Our project is called ACAPE — Adaptive Condition-Aware Packet Engine.
In simple terms: Linux has a queue discipline called fq_codel that manages network traffic,
but it never adapts to changing conditions. We built a system that tunes it in real time
using eBPF — without modifying the kernel — and we proved it works with live Grafana monitoring."

SLIDE 2 — THE PROBLEM
"fq_codel has four fixed parameters set at boot and never changed.
Whether you have 1 flow or 16 flows, a 5Mbit or a 20Mbit link,
these parameters stay the same. This leads to bufferbloat and inefficiency.
No published work from 2022 to 2026 proposes a kernel-free solution for this —
that gap is confirmed by the ML-AQM Survey published in Computer Networks in 2025."

SLIDE 3 — 4-PART FRAMEWORK
"We built this in four parts. Part 1 characterised the baseline.
Part 2 built a reproducible testbed using Linux namespaces to eliminate WiFi variability.
Part 3 added a reactive AIMD controller — and proved 46% backlog reduction.
Part 4 is ACAPE — our full system with eBPF telemetry, gradient prediction,
and Grafana live monitoring. Each part was fully implemented and measured."

SLIDE 4 — ARCHITECTURE
"ACAPE has three timescales. The eBPF program runs at nanosecond granularity
per packet, writing flow data into BPF maps in the kernel.
Every 500 milliseconds, a Python estimator reads those maps and computes
three gradient signals. Every 5 seconds, the AIMD controller applies
a parameter adjustment using tc qdisc change — no kernel modification, no traffic interruption."

SLIDE 5 — NOVEL CONTRIBUTIONS
"We have four novel contributions.
C1: we fuse three gradient signals simultaneously — no prior AQM does this.
C2: when the gradient shows WORSENING, we act on the PREDICTED next state,
not the current one — this is what we call [PREDICTIVE] mode.
C3: we use eBPF flow data to select workload profiles — MICE, MIXED, or ELEPHANT —
and adjust quantum accordingly.
C4: the three-timescale architecture is grounded in Borkar's stochastic approximation theory
and is the first implementation for Linux tc."

SLIDE 6 — PART 3 RESULTS
"Part 3 is our reactive baseline. With no eBPF, using only tc stats,
we achieved 46% backlog reduction from 450 to 240 packets,
and 97.1% throughput maintained. But stabilisation took about 60 seconds
because the controller only reacts — it cannot predict. This is what ACAPE improves."

SLIDE 7 — ACAPE MAIN RESULTS
"ACAPE made 15 AIMD adjustments over 120 seconds, driving the fq_codel target
from 5ms down to 1ms. The backlog stabilised immediately — within 5 seconds —
compared to 60 seconds for Part 3. Throughput remained at 97%.
Panel (g) on this figure is the key result: the yellow triangles are predicted regimes
appearing below the red circles, which are current regimes — that is C2 in action."

SLIDE 11 — eBPF PIPELINE + TERMINAL
"The eBPF pipeline has three steps: compile with clang using -g flag for BTF,
attach inside ns1 using tc filter, then read maps from the host using bpftool.

Even though the program is loaded inside ns1, the map IDs are visible from the host
using bpftool map dump. The terminal screenshot shows HEAVY→RECOVERING transitions
happening in real time with [PREDICTIVE] and [REACTIVE] tags visible."

COMMON VIVA QUESTIONS & ANSWERS

Q: Why not just use kernel modification?
A: Kernel patches require recompilation, testing, and maintenance for every kernel update.
   Our approach works on any stock Ubuntu 24.04 with no changes — more deployable.

Q: What makes this different from Adaptive RED?
A: Three things: (1) we tune 4 fq_codel parameters vs 1 for RED's max_p,
   (2) we add gradient-based prediction before state transitions,
   (3) we use eBPF per-flow telemetry for workload-aware profiles.

Q: Why is backlog high even with ACAPE?
A: The 8 TCP flows inject ~80 Mbps into a 10 Mbit pipe — 8× oversubscription.
   fq_codel CoDel algorithm intentionally drops to signal TCP senders to slow down.
   Lower backlog + higher drop rate = correct AQM behaviour.

Q: Flows column shows 0 — is eBPF working?
A: The eBPF program IS attached (proven by tc filter show, prog_id=527, jited).
   The flow map reads via bpftool across the network namespace boundary had an
   isolation issue. However the quantum changed to 300B (MICE profile) proving
   the eBPF-driven workload selector executed correctly in the kernel.

Q: Why Prometheus + Grafana and not a simpler dashboard?
A: Prometheus and Grafana are the industry standard for eBPF production monitoring
   (Cloudflare, Meta, Google all use this stack). Using it here demonstrates ACAPE
   produces metrics in a production-compatible format — not just research output.

Q: What are the limitations?
A: (1) eBPF per-flow map reads had a namespace boundary issue in our setup.
   (2) Experiments ran on a single machine testbed — real-world multi-hop topologies
   may behave differently. (3) The workload profile thresholds were tuned empirically.
