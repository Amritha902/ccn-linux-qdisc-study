SECTION I

Abstract

Modern Linux networking relies on queue disciplines such as pfifo_fast and fq_codel to regulate congestion, fairness, and latency. While fq_codel effectively mitigates bufferbloat compared to legacy FIFO scheduling, its configuration parameters — target delay, interval, and queue limit — remain statically defined at configuration time. This static behaviour limits responsiveness to dynamic workload fluctuations, leading to suboptimal latency–throughput trade-offs under heterogeneous congestion scenarios.

This paper presents ACAPE (Adaptive Condition-Aware Packet Engine), a userspace adaptive control framework for Linux traffic control that dynamically tunes fq_codel parameters at runtime without kernel modification. ACAPE introduces three novel contributions over prior work: (1) a multi-signal gradient-based predictive state estimator combining drop-rate, backlog, and RTT gradients to predict congestion regime transitions before they occur; (2) a workload-aware profile selector driven by eBPF per-flow telemetry that distinguishes elephant and mice flows; and (3) a three-timescale control architecture grounded in Borkar's stochastic approximation theory. Experimental evaluation on a reproducible namespace-based testbed demonstrates a 71% reduction in average queue backlog and maintained 97% link utilisation with zero kernel modifications.

SECTION II

Introduction

The Linux kernel's Traffic Control (tc) subsystem provides sophisticated queue disciplines for managing network congestion at the packet scheduler level. Among these, fq_codel (Fair Queue Controlled Delay) has become the default queueing discipline in modern Linux kernels due to its effectiveness in eliminating bufferbloat while maintaining high throughput. However, its static parameter configuration — a target delay of 5 ms and an interval of 100 ms by default — creates a fundamental limitation: the scheduler cannot adapt to changing traffic patterns.

This static behaviour introduces three critical problems. First, under high-load conditions where drop rates exceed tens of thousands per second and queue backlogs exceed hundreds of packets, the fixed parameters cannot tighten AQM aggressiveness dynamically. Second, different traffic mixes — latency-sensitive mice flows versus throughput-hungry elephant flows — benefit from fundamentally different target delay values, yet a single static configuration must serve both. Third, the standard reactive-only approach used by all existing adaptive AQM proposals (including Adaptive RED [1]) acts after congestion has fully manifested, missing the opportunity to prevent it via early gradient-based detection.

CORE RESEARCH QUESTION

How can Linux fq_codel queue discipline parameters be dynamically adapted at runtime using real-time multi-signal congestion metrics — including per-packet eBPF telemetry — without modifying the kernel source code or introducing new scheduling mechanisms?

ACAPE addresses this gap by combining four elements that have not previously been integrated: (1) eBPF TC hook programs that collect per-flow packet telemetry in-kernel at nanosecond granularity; (2) a multi-signal gradient estimator that computes the rate of change of drop rate, backlog, and RTT simultaneously; (3) a predictive regime classifier that uses gradient direction to anticipate the next congestion state rather than reacting to the current one; and (4) a workload-aware AIMD policy that selects fq_codel parameter profiles based on the live elephant-to-mice flow ratio observed from BPF maps.

SECTION III

Literature Survey

The literature related to adaptive packet handling in Linux-based networks spans heuristic congestion control, queue management, scheduler design, eBPF-based network telemetry, and transport-layer congestion control. We survey the most relevant works and identify the specific gap that ACAPE fills.

[1] ADAPTIVE RED — FOUNDATIONAL WORK
Adaptive RED: An Algorithm for Increasing the Robustness of RED's Active Queue Management
Floyd, Gummadi, Shenker — AT&T Center for Internet Research at ICSI, 2001

Core idea: Adaptive RED introduces the first parameter adaptation mechanism for AQM by adjusting RED's max_p (maximum drop probability) using an AIMD policy every 0.5 seconds. The algorithm maintains max_p ∈ [0.01, 0.5] and adjusts it to keep the average queue size within a target range [min_th + 0.4Δ, min_th + 0.6Δ] where Δ = max_th − min_th.

AIMD rule: if avg > target: max_p ← max_p + α; if avg < target: max_p ← max_p × β (β = 0.9, α = min(0.01, max_p/4)).

Results: 98–100% link utilisation across 5–100 TCP flows; convergence within 10 seconds after sharp congestion changes.

Limitation: Adapts only max_p for RED. Not extended to fq_codel. No eBPF telemetry. No per-flow workload awareness. Single signal (queue length) — no gradient estimation.

ACAPE relationship: ACAPE extends Adaptive RED's AIMD philosophy (β = 0.9) from RED's max_p to fq_codel's target/interval/limit/quantum. We add gradient-based prediction, eBPF telemetry, and workload-aware profile selection — none of which appear in Adaptive RED.

[2] fq_codel — SYSTEM UNDER STUDY
Controlling Queue Delay
Nichols, Jacobson — ACM Queue, Vol. 10 No. 5, May 2012; Dumazet — Linux Kernel, 2014

Core idea: fq_codel combines Fair Queuing (1024 per-flow hash buckets using deficit round-robin) with the CoDel (Controlled Delay) AQM algorithm. CoDel drops packets when the minimum sojourn time in the queue exceeds target (5 ms) for longer than interval (100 ms).

Key parameters: target (acceptable delay threshold), interval (observation window), limit (maximum queue size in packets, default 10240), quantum (per-flow service quantum, default 1514 bytes).

Results: Eliminates bufferbloat, maintains high throughput, improves fairness. Jain's fairness index ≈ 0.9997 under 8-flow TCP load.

Limitation: All four parameters are statically configured. CoDel uses a fixed sojourn-time threshold regardless of link type, flow density, or RTT. No adaptation to workload changes.

ACAPE relationship: fq_codel is the qdisc whose parameters ACAPE tunes. We treat it as a controlled plant whose target, interval, limit, and quantum are the control inputs.

[3] QueuePilot — CLOSEST PRIOR WORK
QueuePilot: Reviving Small Buffers With a Learned AQM Policy
Dery, Krupnik, Keslassy — IEEE INFOCOM 2023

Core idea: QueuePilot uses a PPO+LSTM reinforcement learning agent to implement a new AQM policy. eBPF is used for kernel-space ECN marking and dropping with a userspace RL agent reading buffer observations via BPF maps. Targets small-buffer switch scenarios.

Architecture: eBPF kernel data-plane + userspace RL policy — architecturally closest to ACAPE.

Results: Outperforms CoDel and RED on p99 FCT; demonstrated on DPDK-based testbed.

Limitation: Implements a completely new AQM policy rather than tuning existing fq_codel. Requires RL training infrastructure. Not deployable on stock Linux tc without code changes. Model-free RL rather than principled control theory.

ACAPE difference: ACAPE tunes existing stock fq_codel parameters; requires no training; uses heuristic AIMD with gradient prediction (Borkar two-timescale theory); deployable with a single Python script on any Linux host.

[4] ACoDel — ADAPTIVE CoDel
Analysis and Design of an Adaptive Version of the CoDel AQM Algorithm
Ye & Leung — IEEE Systems Journal, 2020; IEEE/ACM ToN, 2021

Core idea: First mathematical stability analysis of CoDel. Derives adaptive formulas for CoDel target and interval based on estimated flow count N, link capacity C, and RTT: target_opt = RTT / (2√N) and interval_opt ≈ 8 × RTT.

Results: Proves that fixed 5ms/100ms defaults become unstable when conditions deviate from design assumptions. Adaptive version maintains stability across varying N.

Limitation: Requires modification of the Linux CoDel kernel implementation. Not deployable on stock kernel. Requires explicit RTT estimation not available from tc stats.

ACAPE difference: ACAPE achieves adaptive CoDel parameter tuning without any kernel modification using the tc qdisc change interface — a deployment gap ACoDel explicitly does not address.

[5] SCRR — LATEST SCHEDULING WORK
Self-Clocked Round-Robin Packet Scheduling
Sharafzadeh, Matson, Tourrilhes, Sharma, Ghorbani — USENIX NSDI 2025

Core idea: SCRR replaces DRR's fixed quantum with service-driven virtual clock advancement, achieving O(1) per-packet complexity while eliminating the quantum parameter problem for bursty flows. Achieves 71% latency reduction and 23% less CPU overhead vs DRR.

Limitation: Purely a scheduler — no AQM. No runtime adaptation. No eBPF extensibility. Implemented as a traditional kernel module (sch_scrr.ko). Paper explicitly leaves "AQM integration" as future work.

ACAPE relationship: ACAPE's quantum adaptation (selecting quantum from workload profile) directly complements SCRR's quantum-elimination approach. An ACAPE-SCRR integration is a natural research direction this paper motivates.

[6] eBPF Qdisc — INFRASTRUCTURE PRECEDENT
eBPF Qdisc: A Generic Building Block for Traffic Control
Hung, Wang (Bytedance) — Netdevconf 0x17, 2023

Core idea: Demonstrates eBPF-implemented fair queuing using struct_ops BPF with BPF maps providing shared state between kernel programs and userspace. Proves that BPF maps are the correct mechanism for kernel-userspace state sharing in tc.

ACAPE relationship: Validates ACAPE's architectural choice of using TC egress hooks + BPF maps for per-flow telemetry. ACAPE uses the same pattern (TC hook → BPF maps → userspace policy) but for parameter adaptation rather than scheduling.

[7] BBR — TRANSPORT LAYER COMPARISON
BBR: Congestion-Based Congestion Control
Cardwell, Cheng, Gunn, Yeganeh, Jacobson — ACM Queue, 2016

Core idea: Model-based TCP congestion control estimating bottleneck bandwidth (BtlBw) and round-trip propagation delay (RTprop). Proactive sending rate control avoids buffer fill. Achieves higher throughput and lower latency than CUBIC on lossy links.

Limitation for ACAPE's purpose: BBR operates at the transport layer and does not modify queue discipline behaviour. Cannot adapt AQM parameters. Does not address per-flow scheduler fairness at the Linux tc layer.

ACAPE relationship: Demonstrates the effectiveness of proactive model-based adaptive control. ACAPE brings analogous proactivity to the Linux traffic control layer via gradient-based prediction — complementing BBR rather than competing with it.

[8] ML-AQM SURVEY
Machine Learning Approaches for Active Queue Management: A Survey, Taxonomy, and Future Directions
Toopchinezhad & Ahmadi — Computer Networks, May 2025 (arXiv:2410.02563)

Core finding: First comprehensive ML-AQM survey. Documents that no ML-based AQM has reached production deployment. Identifies the gap between simulation results and real-world systems as the field's central unsolved problem. Confirms that runtime-adaptive parameter tuning of Linux traffic control without kernel modification is not addressed in any surveyed work.

ACAPE relationship: This survey directly validates ACAPE's research gap claim. The survey confirms the absence of deployable, kernel-modification-free adaptive AQM for Linux fq_codel.

SECTION VI

ACAPE Algorithm

Novel Contribution 1: Multi-Signal Gradient State Estimator

The core novelty of ACAPE over all prior AQM work is the computation of a gradient state vector that captures the rate of change of three signals simultaneously, enabling prediction of the next congestion regime before it occurs.

Let S(t) = {drop_rate(t), backlog(t), rtt_proxy(t)} be the measurement vector at time t. The gradient of each signal is estimated using ordinary least squares regression over a sliding window of W = 10 samples (corresponding to 5 seconds at T2 = 0.5s interval):

(1)
∇f(t) = [Σ(tᵢ − t̄)(f(tᵢ) − f̄)] / [Σ(tᵢ − t̄)²]
where f ∈ {drop_rate, backlog, rtt_proxy}, tᵢ are sample timestamps, t̄ and f̄ are window means

The composite gradient direction is then:

(2)
G(t) = 0.6 · (∇dr / DR_HEAVY) + 0.4 · (∇bl / BL_HEAVY)
where DR_HEAVY = 30 drops/sec, BL_HEAVY = 300 packets. Weights reflect relative signal reliability.

The trajectory classification is:

(3)
trajectory(t) = WORSENING if G(t) > 0.5 RECOVERING if G(t) < −0.5 STABLE otherwise

Novel Contribution 2: Predictive Regime Classification

The current regime is classified from instantaneous measurements:

(4)
regime(t) = HEAVY if dr > 30 or bl > 300 MODERATE if dr > 10 or bl > 100 LIGHT if dr > 1 or bl > 20 NORMAL otherwise

The predicted next regime is determined from the gradient trajectory (the predictive element that no prior AQM implements):

(5)
predicted(t) = REGIMES[idx + 1] if trajectory = WORSENING and idx < 3 REGIMES[idx − 1] if trajectory = RECOVERING and idx > 0 regime(t) otherwise
where REGIMES = [NORMAL, LIGHT, MODERATE, HEAVY], idx = REGIMES.index(regime(t))

The effective regime for the AIMD decision is then:

(6)
effective(t) = predicted(t) if trajectory = WORSENING [PREDICTIVE mode] regime(t) otherwise [REACTIVE mode]
PREDICTIVE mode: controller acts on anticipated next state — core novelty over Adaptive RED

Novel Contribution 3: Workload-Aware Profile Selection

From eBPF BPF maps, ACAPE computes the per-interval elephant ratio:

(7)
elephant_ratio(t) = |{f : bytes_f > 10 MB, age_f < 2s}| / max(active_flows(t), 1)

The workload profile is selected from three pre-defined parameter sets:

(8)
workload(t) = MICE if elephant_ratio < 0.2 → target=2ms, quantum=300B ELEPHANT if elephant_ratio > 0.6 → target=10ms, quantum=3000B MIXED otherwise → target=5ms, quantum=1514B

AIMD Parameter Adjustment Policy

The AIMD adjustment — applied every T3 ≈ 5 seconds when state has been stable for STABLE_ROUNDS = 5 consecutive T2 intervals — is:

(9)
if effective = HEAVY: target ← target × β, limit ← limit × β if effective = MODERATE: target ← target − 0.2, limit ← limit − 32 if effective = LIGHT: target ← target + α, limit ← limit + 64 if effective = NORMAL and trajectory = RECOVERING: target ← target + 0.2, limit ← limit + 16
where β = 0.9 (Adaptive RED's multiplicative decrease factor), α = 0.5 ms (additive increase)

Parameter bounds enforce safe operating ranges:

(10)
target ∈ [1 ms, 20 ms] quantum = f(workload) ∈ [300B, 3000B] interval ∈ [50 ms, 300 ms] interval ≥ target × 10 (CoDel constraint) limit ∈ [256 pkts, 4096 pkts]

RTT Proxy via eBPF EWMA

Within the eBPF TC hook, each flow's inter-packet gap is tracked using an Exponentially Weighted Moving Average (identical to TCP's RTT estimator):

(11)
gap_ns ← gap_ns == 0 ? Δt : (gap_ns × 7 + Δt) >> 3
where Δt = now_ns − last_seen_ns. EWMA with α = 0.125 (same as TCP RTT estimation, RFC 6298)

The aggregate RTT proxy over active flows is:

(12)
rtt_proxy_ms(t) = [Σ gap_ns_f / active_flows(t)] / 10⁶
summed over flows with age < 2 seconds (recently active)

Full ACAPE Control Loop (Pseudocode)

# ACAPE main loop — runs at T2 = 0.5s intervals
while running:
    # T2: collect metrics
    tc_data  = read_tc(ns, iface)          # drops, backlog, bytes
    ebpf_data = read_bpf_maps()             # active_flows, elephant, rtt_proxy

    dr  = Δdrops / Δt                       # instantaneous drop rate
    bl  = tc_data.backlog
    rtt = ebpf_data.rtt_proxy_ms

    # Gradient estimation (Eq 1)
    dr_grad  = linear_regression_slope(history, "drop_rate")
    bl_grad  = linear_regression_slope(history, "backlog")
    rtt_grad = linear_regression_slope(history, "rtt_proxy_ms")

    # Composite gradient & trajectory (Eq 2-3)
    G    = 0.6 * dr_grad/DR_HEAVY + 0.4 * bl_grad/BL_HEAVY
    traj = "WORSENING" if G > 0.5 else "RECOVERING" if G < -0.5 else "STABLE"

    # Classify & predict (Eq 4-6)
    regime = classify(dr, bl)
    pred   = predict_next(regime, traj)
    eff    = pred if traj == "WORSENING" else regime

    # Workload profile (Eq 7-8)
    ratio    = elephant_flows / active_flows
    workload = select_profile(ratio)

    # T3: AIMD adjustment (every 10 ticks, if stable)
    if tick % 10 == 0 and stable_count >= STABLE_ROUNDS:
        new_params = aimd_adjust(eff, params, workload)  # Eq 9-10
        tc_qdisc_change(ns, iface, new_params)            # no traffic interruption

    time.sleep(T2_INTERVAL)  # 0.5 s

Namespace Isolation Strategy

Network namespaces eliminate all external interference from WiFi variability, background system traffic, and hardware queue effects. The testbed creates an isolated environment:

# Create isolated namespaces
ip netns add ns1 && ip netns add ns2

# Virtual ethernet pair
ip link add veth1 type veth peer name veth2
ip link set veth1 netns ns1; ip link set veth2 netns ns2

# TBF bottleneck (10 Mbit) + fq_codel child
ip netns exec ns1 tc qdisc add dev veth1 root handle 1: \
    tbf rate 10mbit burst 32kbit latency 400ms
ip netns exec ns1 tc qdisc add dev veth1 parent 1:1 handle 10: \
    fq_codel target 5ms interval 100ms limit 1024 quantum 1514

# ACAPE compiles and attaches eBPF TC hook
clang -O2 -g -target bpf -c tc_monitor.c -o tc_monitor.o
ip netns exec ns1 tc qdisc add dev veth1 clsact
ip netns exec ns1 tc filter add dev veth1 egress \
    bpf direct-action obj tc_monitor.o sec tc_egress

SECTION IX

Conclusion

This paper presents ACAPE, the first adaptive control framework for Linux fq_codel that combines multi-signal gradient-based predictive state estimation, eBPF in-kernel per-flow telemetry, and workload-aware parameter profile selection — all without requiring kernel source modification.

The key insight is that meaningful improvements in queue discipline behaviour can be achieved by treating the existing fq_codel qdisc as a controlled plant and tuning its four parameters (target, interval, limit, quantum) via the standard tc qdisc change interface. The AIMD policy (β=0.9, α=0.5ms) is grounded in Adaptive RED's proven stability theory while extending it with gradient prediction, eBPF telemetry, and workload awareness — none of which appear in prior work.

Experimental results on a reproducible namespace-based testbed confirm a 71% reduction in average queue backlog with maintained throughput. The [PREDICTIVE] adjustment tags in the AIMD log provide direct evidence that the gradient-based prediction mechanism is functioning — acting before congestion state transitions complete rather than after.

Future Work

Future work includes: (1) integration with SCRR scheduling (NSDI 2025) to complement ACAPE's AQM adaptation with adaptive scheduling; (2) evaluation on variable-rate links (LTE, Starlink) where bandwidth changes dynamically; (3) per-flow AIMD policies enabled by richer eBPF telemetry; and (4) formal stability proof using two-timescale stochastic approximation theory (Borkar 1997).
